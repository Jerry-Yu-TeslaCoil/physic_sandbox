# 主要结构

- ### 设计原则
  - 所有组件是可信的，可以自由访问事件总线或者发布消息
  - 所有物理效果尽量实现，综合考虑准确性和性能，不要求极端准确
  - 所有事件尽可能早响应，平衡时效性、帧调度和模块耦合度，不要求即时处理
  - 模拟优于实现

- ### 机制结构
  - 物理分析器PhysicAnalyzer
     - 物理分析层
     - 碰撞检测层
     - 碰撞约束生成层
     - 约束求解层
        - 顺序脉冲法
     - 物理更新层
  - 事件总线EventBus
     - 维护一个映射表，映射事件标签以及对应监听器列表
     - 监听器注册到事件总线时说明自身要监听的事件标签集，供事件总线将其添加到对应监听器列表
     - 任何组件都能在任意时刻直接调用事件总线发布消息（因为此处假定所有组件都是可信的）
     - 事件总线将消息发布时间记录为实际发生时的时间戳
     - 在计算帧前期分发对应时间段内的所有事件
     - 在计算帧完毕后消费掉时间段内的所有事件
  - 画面渲染器Renderer
  - 输入控制器Controller
     - 点击，进入退出，按下抬起等为操作，一次触发缓冲区增加一个事件，由计算帧计算后直接消费
     - 将用户输入操作解释为具体语义事件
     - 用户输入操作应记录时间戳并携带到语义事件，供对应计算帧消费（添加时间戳由事件总线发布方法完成）
     - 完成操作创建和解释后，将得到的语义事件发布到事件总线以供监听
  - 帧调度管理器FrameManager
     - 维护游戏时间线：
        - 记录游戏实际运行时间戳并暴露
  - 生命周期管理器LifeCycleManager
     - 持有所有元素的引用，并管理元素和组件的生命周期
     - 添加组件或添加新元素时，记录组件的生效阶段于映射表，在对应阶段调用
     - 元素不再生效时，解除组件在映射表中的存储，并取消对对象的引用，让GC自动释放

- ### 每帧阶段处理
  1. 每次画面帧由帧调度管理器发起，调用生命周期管理器执行刷新，一个画面帧执行多次计算帧
     - 根据时间记录计算两次画面帧时间差dt，记录当前时间t用于下次dt计算
     - 计算当前帧时长。固定时间步长t'，根据t，dt和t'计算本次画面帧应执行c次计算帧（尾部多余时间由下次处理）
     - **举例**：当前时间1.4秒，dt为0.016秒，t'为0.007秒。
        - 此帧开始时t2为1.384秒，对0.007秒余数为0.005秒，即上一帧剩余未处理时间
        - 即此帧第一次计算帧时间为1.389秒，其后1.396秒为下一需要的计算帧，之后即超过当前时间。因此本画面帧计算2次计算帧
  2. 计算c次计算帧内处理
     1. 总线分发计算帧内事件
     2. 物理分析及元素合冲量计算
     3. 碰撞检测
     4. 碰撞约束生成
     5. 迭代约束求解
     6. 物理更新
     7. 事件移除
     8. 自销毁组件处理（比如碰撞等瞬时约束组件）
  3. 画面渲染

- ### 对象结构
  - 元素Element定义容器
     - addComponent(Component component): 添加组件
     - removeComponent(Component component): 移除组件
     - update(): 执行对象更新
  - 组件Component定义行为
     - 组件维护一个自身的执行阶段，如渲染组件在画面渲染阶段执行更新，物理组件在物理分析层，约束组件在约束求解层，等等
     - **组件的通用方法**
        - update(): 执行组件更新，添加力、位移，发布事件，渲染画面，监听输入
        - setActivate(boolean isActivate): 设置是否生效
        - setUpdateLayer(): 设置更新阶段
        - autoDisposable(): 设置组件是否执行后的组件销毁阶段自动销毁（用于一个计算帧内的瞬时影响，如碰撞）
     - 组件具体实现
        - Renderer
           - 渲染物体画面
           - 更新于画面渲染阶段
        - Controller
           - 监听用户输入并对物体创建冲量或执行操作
           - 更新于物理分析及元素合冲量计算阶段
