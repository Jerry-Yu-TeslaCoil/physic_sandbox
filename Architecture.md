# 主要结构

- ### 设计原则
  - 所有组件是可信的，可以自由访问事件总线或者发布消息
  - 所有物理效果尽量实现，综合考虑准确性和性能，不要求极端准确
  - 所有事件尽可能早响应，平衡时效性、帧调度和模块耦合度，不要求即时处理
  - 模拟优于实现

- ### 机制结构
  - 输入控制器Controller
    - 点击，进入退出，按下抬起等为操作，一次触发缓冲区增加一个事件，由计算帧计算后直接消费
    - 将用户输入操作解释为具体语义事件
    - 用户输入操作应记录时间戳并携带到语义事件，供对应计算帧消费（添加时间戳由事件总线发布方法完成）
    - 完成操作创建和解释后，将得到的语义事件发布到事件总线以供监听
  - 帧调度管理器FrameManager
    - 维护游戏时间线：
      - 记录游戏实际运行时间戳并暴露
  - 生命周期管理器LifeCycleManager
    - 持有所有元素的引用，并管理元素和组件的生命周期
    - 添加组件或添加新元素时，记录组件的生效阶段于映射表，并将组件挂载到对应执行器
    - 元素不再生效时，解除组件在映射表中的存储以及执行器的挂载，并取消对对象的引用，让GC自动释放
  - **执行器**
     - 物理分析器PhysicAnalyzer
       - 物理分析层
     - 物理更新器PhysicUpdater
       - 物理更新层
     - 碰撞分析器ColliderAnalyzer
       - 碰撞检测层
       - 碰撞约束生成层
     - 约束求解器ConstraintSolver
       - 约束求解层
          - 顺序脉冲法
     - 画面渲染器Renderer
     - 事件总线EventBus
       - 维护一个映射表，映射事件标签以及对应监听器列表
       - 监听器注册到事件总线时说明自身要监听的事件标签集，供事件总线将其添加到对应监听器列表
       - 任何组件都能在任意时刻直接调用事件总线发布消息（因为此处假定所有组件都是可信的）
       - 事件总线将消息发布时间记录为实际发生时的时间戳
       - 在计算帧前期分发对应时间段内的所有事件
       - 在分发完毕后消费掉时间段内的所有事件
       - v1.0 不用显式字符串标签，而是改用Listener自带泛型说明监听的Event子类类型

- ### 对象结构
  - 元素GameObject定义容器
    - addComponent(Component component): 添加组件
    - removeComponent(Component component): 移除组件
    - update(): 执行对象更新
  - 组件Component定义行为
    - 组件维护一个自身的执行阶段，如渲染组件在画面渲染阶段执行更新，物理组件在物理分析层，约束组件在约束求解层，等等
    - **组件的通用方法**
      - update(): 执行组件更新，添加力、位移，发布事件，渲染画面，监听输入
      - setActivate(boolean isActivate): 设置是否生效
      - setUpdateLayer(): 设置更新阶段
        - *考虑用注解完成此功能*
      - autoDisposable(): 设置组件是否执行后的组件销毁阶段自动销毁（用于一个计算帧内的瞬时影响，如碰撞）
    - 组件具体实现
      - Transform
        - 管理物体位置、旋转、缩放、当前速度等信息
        - 存储物体所受冲量信息
        - 根据信息更新下一计算帧状态
        - 挂载于物理更新器
      - Renderer
        - 渲染物体画面
        - 挂载于画面渲染器
      - Controller
        - 监听用户输入并对物体创建冲量或执行操作
        - 挂载于物理分析器
      - Collider
        - 检测碰撞并发送事件
        - 挂载于碰撞分析器
      - Constraint
        - 对元素进行约束
        - 挂载于约束求解器

- ### 每帧阶段处理
  1. 画面帧开始
     - 由帧调度管理器发起，一个画面帧执行多次计算帧
     - 根据时间记录计算两次画面帧时间差dt，记录当前时间t用于下次dt计算
     - 计算当前帧时长。固定时间步长t'，根据t，dt和t'计算本次画面帧应执行c次计算帧（尾部多余时间由下次处理）
     - **举例**：当前时间1.4秒，dt为0.016秒，t'为0.007秒。
        - 此帧开始时t2为1.384秒，对0.007秒余数为0.005秒，即上一帧剩余未处理时间
        - 即此帧第一次计算帧时间为1.389秒，其后1.396秒为下一需要的计算帧，之后即超过当前时间。因此本画面帧计算2次计算帧
  2. 计算c次计算帧
     - 帧调度完成后，由帧调度管理器对每一计算帧的每一执行层，调用各自的执行器发起
     - 帧调度管理器应为执行器提供当前计算帧时间戳
     - 在一计算帧内分阶段调用对应组件更新:
        1. 总线分发当前计算帧内事件
           - **调用事件总线执行**
           - 筛选出所有在计算帧结束前的事件
           - 对于每个应在当前计算帧处理的事件，将其分发给对应监听器
           - 对每个监听器更新
           - v1.0 对每个监听器组件，包含两个方法: 监听器的onEvent和组件的update
           - v1.0 onEvent只负责消息的获取，具体处理逻辑放update
        2. 物理计算
           - **调用物理分析器执行**
           - 物理分析及元素合冲量计算
           - 一般组件的更新操作大部分在此完成，如添加重力冲量、外力冲量
        3. 物理更新
           - **调用物理更新器执行**
           - 更新物体的位移
        4. 碰撞约束计算
           - **调用碰撞分析器执行**
           1. 碰撞检测
              - 更新碰撞体相关组件
              - 检测碰撞并发布碰撞事件
              - 碰撞分组存储
           2. 碰撞约束生成
              - 根据碰撞分组对元素添加碰撞约束
        5. 约束求解
           - **调用约束求解器执行**
           - 执行约束的迭代求解
           - 修正位置以满足约束
        6. 帧结束清理
           - 事件移除
           - 自销毁组件处理（比如碰撞等瞬时约束组件）
           - 其它自定义清理
  3. 画面渲染。此画面帧计算全部完成后，由帧调度管理器调用画面渲染器发起，更新渲染组件
